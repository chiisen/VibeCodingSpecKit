Spec-Driven Development 重新定義開發體驗

GitHub Spec Kit 讓規格成為可執行的核心產物。從想法到部署，透過自動化 CLI 與 AI 協作流程，專注在打造真正重要的產品情境。

# 規格為真。程式碼跟隨規格。

Spec-Driven Development 是什麼？

Spec-Driven Development（SDD）顛覆傳統先寫程式後看規格的模式。規格不再只是一次性的文件，而是驅動開發的真實來源：藉由可執行的 specification，直接產出符合情境的實作。Spec Kit 提供工具鏈與最佳實踐，讓團隊以規格為中心展開協作，確保需求、設計與交付一致。

為什麼選擇 SDD

  • 以規格為單一真實來源，減少溝通落差。
  • 自動化產生程式碼，將人力專注在產品差異化。
  • 規格、流程、實作同時演進，加速迭代。

Spec Kit 提供什麼

  • Specify CLI：初始化、驗證、執行規格。
  • 分層章程：Constitution、核心指令與工作流程。
  • AI Agents 指南：與 Claude、Copilot 等工具協作。

立即可用的成果

  • 一鍵建立規格倉庫與專案骨架。
  • 嚴謹驗證指令確保規格品質。
  • 逐步開發任務分解與自動化實施。

快速入門：3 個步驟啟動

使用 Specify CLI 可以在幾分鐘內準備好 SDD 工作環境。建議透過 uv tool 安裝以便於版本管理。

01｜安裝指定工具

持久安裝（建議）：

uv tool install specify-cli \
  --from git+https://github.com/github/spec-kit.git

一次性使用：

uvx --from git+https://github.com/github/spec-kit.git \
  specify init <PROJECT_NAME>

02｜初始化專案

建立全新 spec 專案骨架：

specify init my-product-spec
  • 自動生成 constitution 與核心流程。
  • 建立規格資料夾、模板與工具設定。

03｜品質檢查

確保規格與流程一致：

specify check
  • 驗證章程、規格與任務清單。
  • 及早發現缺漏與衝突。

核心哲學

Spec Kit 以三大支柱構成一套系統化的開發方法，確保團隊在規格驅動下保持靈活、快速且高品質。

Constitution 章程

  • 定義專案的核心原則、語彙與節奏。
  • 建立 AI 協作守則，避免偏離目標。

Core Commands

  • /specify：釐清需求與約束。
  • /plan：產出實作藍圖與風險清單。
  • /tasks：拆解為可執行任務，支援 TDD/BDD。

AI Co-pilot

  • 明確指示 Claude Code、Copilot、Gemini CLI 的角色。
  • 讓 AI 負責重複性工作，人類聚焦產品價值。

三階段開發流程

SDD 將開發流程拆分為三個循環階段，從提案、實作到封存都與規格緊密連結。

Stage 1｜Creating Changes

針對新功能或大幅調整建立 proposal，撰寫變更動機、影響與 spec delta。使用 openspec validate --strict 確保規格正確。

Stage 2｜Implementing Changes

依據核準的 proposal 與 tasks.md 執行實作。每一步都有 AI 協作提示，遵循章程與測試策略。

Stage 3｜Archiving Changes

部署完成後，在變更存檔中保留歷史，維持 specs 為唯一真實來源，並為後續改善留下 traceability。

支援的 AI 協作夥伴

Spec Kit 內建針對多種 AI 工具的操作指引。透過一致的指令語彙與檢查清單，降低溝通與交接成本。

Claude Code

  • 協助審閱章程、調整計畫與驗證任務。
  • 可自動生成 PR 描述與檢查清單。

GitHub Copilot

  • 配合規格產出樣板碼。
  • 在 IDE 中提供即時建議與測試片段。

Gemini CLI / 其他 Agents

  • 以同一套流程指令進行協作。
  • 善用 `openspec` 工具鏈進行驗證與封存。
